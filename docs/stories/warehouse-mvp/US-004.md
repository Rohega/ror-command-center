# US-004: Gestión de ubicaciones

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Prioridad:** P0  
**Dependencias:** US-003

---

## Objetivo

Definir ubicaciones físicas dentro de cada almacén para saber dónde se almacena cada producto y optimizar recorridos de picking.

## Usuario

**Administrador** (`admin`)

## Descripción

Como administrador, quiero crear ubicaciones dentro de un almacén (pasillo, estante, posición), para asignar stock a posiciones concretas y generar listas de picking ordenadas.

## Criterios de aceptación

- [ ] Puedo crear ubicaciones asociadas a un almacén
- [ ] Cada ubicación tiene código legible único dentro del almacén (ej. `A-01-03`)
- [ ] Campos: pasillo (`aisle`), estante (`rack`), posición (`position`) y código compuesto
- [ ] Puedo desactivar ubicación sin stock
- [ ] No puedo desactivar ni eliminar ubicación con `quantity_on_hand > 0`
- [ ] Listado filtrable por almacén y estado

## Reglas de negocio

1. Código único por `(warehouse_id, code)`.
2. Ubicación pertenece a exactamente un almacén.
3. Ubicación inactiva no acepta nuevas asignaciones de stock.
4. El código puede generarse automáticamente desde aisle/rack/position o ingresarse manualmente (decisión implementación: auto desde componentes).

## Casos borde

| Caso | Comportamiento esperado |
|------|-------------------------|
| Stock en ubicación al desactivar | Bloquear desactivación |
| Dos ubicaciones mismo código en almacenes distintos | Permitido |
| Ubicación sin stock, con movimientos históricos | Permitir desactivar |
| Cambiar código con stock presente | Rechazado en MVP |

## Notas técnicas preliminares

- Modelo `Location`: `warehouse_id`, `code`, `aisle`, `rack`, `position`, `active`
- Índice único compuesto `(warehouse_id, code)`
- Callback o servicio para generar `code` = `"#{aisle}-#{rack}-#{position}"`
- API: `GET/POST/PATCH /api/warehouses/:id/locations`
- Picking ordenará por `aisle`, `rack`, `position` lexicográficamente
