# US-002: Gestión de productos

**Spec:** [warehouse-mvp](../../specs/warehouse-mvp.md)  
**Prioridad:** P0  
**Dependencias:** US-001

---

## Objetivo

Mantener el catálogo maestro de productos que la distribuidora mueve entre almacenes, con identificación por SKU y búsqueda flexible sin depender de código de barras.

## Usuario

**Administrador** (`admin`)

## Descripción

Como administrador, quiero crear y editar productos con SKU, nombre, categoría y unidad de medida, y opcionalmente un código de barras, para que operarios y supervisores puedan identificarlos en recepción, salida y consultas.

## Criterios de aceptación

- [ ] Puedo crear un producto con: SKU (obligatorio), nombre, categoría, unidad de medida
- [ ] Unidad de medida solo admite: `unidad`, `caja`, `paquete`, `kg`, `litro`
- [ ] Puedo asignar `barcode` opcional; si se informa, debe ser único en el sistema
- [ ] No puedo duplicar SKU
- [ ] Puedo desactivar un producto; no aparece en nuevas órdenes de recepción/salida/transferencia
- [ ] Puedo buscar productos por SKU, nombre (parcial) o categoría
- [ ] El listado muestra SKU, nombre, categoría, unidad, estado y barcode si existe

## Reglas de negocio

1. SKU único global e inmutable tras creación (no editable en MVP).
2. `barcode` nullable; si presente, único global.
3. Producto inactivo: stock existente permanece; no nuevas operaciones.
4. Sin conversiones entre unidades: toda cantidad opera en `unit_type` del producto.
5. Categoría debe estar activa al crear producto nuevo.

## Casos borde

| Caso | Comportamiento esperado |
|------|-------------------------|
| Barcode vacío vs null | Tratados como sin barcode |
| Dos productos con mismo barcode | Rechazado en validación |
| Desactivar producto con stock > 0 | Permitido con advertencia |
| Búsqueda por nombre con acentos | Búsqueda case-insensitive, normalización básica |
| SKU con espacios al inicio/fin | Trim automático |

## Notas técnicas preliminares

- Modelo `Product`: `sku`, `name`, `category_id`, `unit_type` (enum), `barcode` (nullable, unique), `active`
- Validación enum `unit_type` en modelo y API
- Preparar `has_many :external_references` para v2 Odoo
- Índices: `sku` (unique), `barcode` (unique, nulls allowed), `name` (fulltext o índice para LIKE)
- API: `GET/POST/PATCH /api/products` con filtros `q`, `category_id`, `active`
